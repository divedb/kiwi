349 // Attempts to take an exclusive write lock on the file. Returns immediately
350 // (i.e. does not wait for another process to unlock the file). If the lock
351 // was obtained, the result will be FILE_OK. A lock only guarantees
352 // that other processes may not also take a lock on the same file with the
353 // same API - it may still be opened, renamed, unlinked, etc.
354 //
355 // Common semantics:
356 // * Locks are held by processes, but not inherited by child processes.
357 // * Locks are released by the OS on file close or process termination.
358 // * Locks are reliable only on local filesystems.
359 // * Duplicated file handles may also write to locked files.
360 // Windows-specific semantics:
361 // * Locks are mandatory for read/write APIs, advisory for mapping APIs.
362 // * Within a process, locking the same file (by the same or new handle)
363 // will fail.
364 // POSIX-specific semantics:
365 // * Locks are advisory only.
366 // * Within a process, locking the same file (by the same or new handle)
367 // will succeed. The new lock replaces the old lock.
368 // * Closing any descriptor on a given file releases the lock.
369 Error Lock(LockMode mode);
370
371 // Unlock a file previously locked.
372 Error Unlock();

383#if BUILDFLAG(IS_WIN)
384 // Sets or clears the DeleteFile disposition on the file. Returns true if
385 // the disposition was set or cleared, as indicated by |delete_on_close|.
386 //
387 // Microsoft Windows deletes a file only when the DeleteFile disposition is
388 // set on a file when the last handle to the last underlying kernel File
389 // object is closed. This disposition is be set by:
390 // - Calling the Win32 DeleteFile function with the path to a file.
391 // - Opening/creating a file with FLAG_DELETE_ON_CLOSE and then closing all
392 // handles to that File object.
393 // - Opening/creating a file with FLAG_CAN_DELETE_ON_CLOSE and subsequently
394 // calling DeleteOnClose(true).
395 //
396 // In all cases, all pre-existing handles to the file must have been opened
397 // with FLAG_WIN_SHARE_DELETE. Once the disposition has been set by any of the
398 // above means, no new File objects can be created for the file.
399 //
400 // So:
401 // - Use FLAG_WIN_SHARE_DELETE when creating/opening a file to allow another
402 // entity on the system to cause it to be deleted when it is closed. (Note:
403 // another entity can delete the file the moment after it is closed, so not
404 // using this permission doesn't provide any protections.)
405 // - Use FLAG_DELETE_ON_CLOSE for any file that is to be deleted after use.
406 // The OS will ensure it is deleted even in the face of process termination.
407 // Note that it's possible for deletion to be cancelled via another File
408 // object referencing the same file using DeleteOnClose(false) to clear the
409 // DeleteFile disposition after the original File is closed.
410 // - Use FLAG_CAN_DELETE_ON_CLOSE in conjunction with DeleteOnClose() to alter
411 // the DeleteFile disposition on an open handle. This fine-grained control
412 // allows for marking a file for deletion during processing so that it is
413 // deleted in the event of untimely process termination, and then clearing
414 // this state once the file is suitable for persistence.
415 bool DeleteOnClose(bool delete_on_close);
