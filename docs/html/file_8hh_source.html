326 // Attempts to take an exclusive write lock on the file. Returns immediately
327 // (i.e. does not wait for another process to unlock the file). If the lock
328 // was obtained, the result will be FILE_OK. A lock only guarantees
329 // that other processes may not also take a lock on the same file with the
330 // same API - it may still be opened, renamed, unlinked, etc.
331 //
332 // Common semantics:
333 // * Locks are held by processes, but not inherited by child processes.
334 // * Locks are released by the OS on file close or process termination.
335 // * Locks are reliable only on local filesystems.
336 // * Duplicated file handles may also write to locked files.
337 // Windows-specific semantics:
338 // * Locks are mandatory for read/write APIs, advisory for mapping APIs.
339 // * Within a process, locking the same file (by the same or new handle)
340 // will fail.
341 // POSIX-specific semantics:
342 // * Locks are advisory only.
343 // * Within a process, locking the same file (by the same or new handle)
344 // will succeed. The new lock replaces the old lock.
345 // * Closing any descriptor on a given file releases the lock.
346 Error Lock(LockMode mode);
347
348 // Unlock a file previously locked.
349 Error Unlock();

372 // Sets or clears the DeleteFile disposition on the file. Returns true if
373 // the disposition was set or cleared, as indicated by |delete_on_close|.
374 //
375 // Microsoft Windows deletes a file only when the DeleteFile disposition is
376 // set on a file when the last handle to the last underlying kernel File
377 // object is closed. This disposition is be set by:
378 // - Calling the Win32 DeleteFile function with the path to a file.
379 // - Opening/creating a file with FLAG_DELETE_ON_CLOSE and then closing all
380 // handles to that File object.
381 // - Opening/creating a file with FLAG_CAN_DELETE_ON_CLOSE and subsequently
382 // calling DeleteOnClose(true).
383 //
384 // In all cases, all pre-existing handles to the file must have been opened
385 // with FLAG_WIN_SHARE_DELETE. Once the disposition has been set by any of the
386 // above means, no new File objects can be created for the file.
387 //
388 // So:
389 // - Use FLAG_WIN_SHARE_DELETE when creating/opening a file to allow another
390 // entity on the system to cause it to be deleted when it is closed. (Note:
391 // another entity can delete the file the moment after it is closed, so not
392 // using this permission doesn't provide any protections.)
393 // - Use FLAG_DELETE_ON_CLOSE for any file that is to be deleted after use.
394 // The OS will ensure it is deleted even in the face of process termination.
395 // Note that it's possible for deletion to be cancelled via another File
396 // object referencing the same file using DeleteOnClose(false) to clear the
397 // DeleteFile disposition after the original File is closed.
398 // - Use FLAG_CAN_DELETE_ON_CLOSE in conjunction with DeleteOnClose() to alter
399 // the DeleteFile disposition on an open handle. This fine-grained control
400 // allows for marking a file for deletion during processing so that it is
401 // deleted in the event of untimely process termination, and then clearing
402 // this state once the file is suitable for persistence.
403 bool DeleteOnClose(bool delete_on_close);

430 // This function can be used to augment `flags` with the correct flags
431 // required to create a File that can be safely passed to an untrusted
432 // process. It must be called if the File is intended to be transferred to an
433 // untrusted process, but can still be safely called even if the File is not
434 // intended to be transferred.
435 static constexpr uint32_t AddFlagsForPassingToUntrustedProcess(
436 uint32_t flags) {
437 if (flags & File::FLAG_WRITE || flags & File::FLAG_APPEND ||
438 flags & File::FLAG_WRITE_ATTRIBUTES) {
439 flags |= File::FLAG_WIN_NO_EXECUTE;
